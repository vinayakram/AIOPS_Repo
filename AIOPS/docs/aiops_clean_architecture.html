AIOps Clean Architecture

Project: POCDEMO1 Application: SampleAgent / sample-agent Observability: Prometheus, traces, RCA agents Updated: 17 April 2026

Purpose

This document explains how the AIOps proof of concept works as a clean production-style architecture. It shows the technical components, how telemetry moves through the system, what scenarios are already covered, and where the current limitations are.

The central idea is simple: the business application emits telemetry, the AIOps platform detects issues, the RCA service gathers evidence from metrics and traces, and remediation produces either an executable code/config change or an operator plan when infrastructure permissions are required.

Telemetry Flow

  1. User or test traffic calls SampleAgent /api/query.
  2. SampleAgent records request status, duration, concurrency, article counts, and LLM call outcomes.
  3. SampleAgent exposes those values at /metrics.
  4. Prometheus scrapes http://10.169.91.16:8002/metrics as job="sample-agent".
  5. SampleAgent also emits request traces and span errors to the trace path used by the AIOps/RCA flow.
  6. AIopsTelemetry detects issues such as p95 latency under load or LLM provider throttling.
  7. RCA calls Prometheus query_range and retrieves trace/span evidence for the same incident window.
  8. Remediation uses RCA output to suggest or implement code, config, or infrastructure changes.

Covered Scenarios

Concurrent Users / Load Degradation

Many users call SampleAgent at once. p95 latency, error count, and max concurrency are captured through Prometheus metrics. AIOps can raise an NFR latency issue and RCA can recommend scaling, resource tuning, or backpressure.

CoveredNeeds pod validation for production realism

Azure OpenAI Rate Limit

The low-rate Azure deployment is called rapidly. Provider-side failures such as HTTP 429 are captured as failed LLM spans and Prometheus LLM counters.

CoveredReal provider call path

RCA Evidence Quality

RCA now separates trace evidence from Prometheus metrics and should avoid claiming Prometheus proved something when a query returned no samples.

Remediation Plan

Remediation can propose code, configuration, or infrastructure steps. When Codex lacks cluster permissions, it produces a user-executable plan instead of pretending the change was applied.

What Prometheus Does and Does Not Do

Prometheus stores numeric metrics, not full application log lines. The RCA display may show rows with source=prometheus, but those are formatted evidence rows generated by the investigation service from Prometheus query results.

Evidence Type System of Record Example
Numeric metric Prometheus sample_agent_llm_requests_total{status="rate_limited"} 2
Request-level error text Trace/span store openai_generation span with Azure HTTP 429 error details
Raw application logs Not yet implemented as a log backend Recommended future addition: Loki or OpenTelemetry Collector logs pipeline

Kubernetes Pod Hosting Path

SampleAgent can be hosted as a Kubernetes pod. That is the right next step for validating concurrent-user failure in a production-like environment.

  • Create a Deployment for SampleAgent with CPU and memory requests/limits.
  • Expose it with a Service on port 8002.
  • Configure Prometheus scrape annotations or a ServiceMonitor for /metrics.
  • Run load from k6, Locust, hey, or a Kubernetes Job.
  • Validate that Prometheus captures pod-level and app-level pressure: CPU, memory, p95 latency, errors, and in-flight requests.
  • Use RCA to recommend horizontal scaling, resource tuning, worker tuning, caching, queueing, or backpressure.

Recommended Next Production Code-Fix Scenario

The best next production code-fix scenario is provider-aware LLM throttling remediation. It is more realistic than the special-character case and maps directly to a common GenAI production failure.

Incident RCA Cause Useful Production Fix
Azure OpenAI returns HTTP 429 during traffic burst Deployment request/token quota exceeded or burst limit reached Add bounded concurrency, provider-aware exponential backoff, Retry-After handling, timeout budgets, and user-safe fallback
SampleAgent p95 latency rises under concurrent load RAG pipeline performs expensive synchronous work per request Add backpressure, queue limits, cache reuse, worker tuning, and clearer saturation metrics

Current Limitations

  • Prometheus metrics are real, but raw application logs are not yet stored in a log backend.
  • Process-local gauges reset when SampleAgent restarts.
  • For multiple pods, PromQL must aggregate across instances intentionally.
  • Provider rate-limit behavior depends on Azure's actual throttling, which may allow short bursts.
  • GitHub PR/push automation depends on credentials and repository access.
  • Infrastructure remediation requires Kubernetes or cloud permissions.
