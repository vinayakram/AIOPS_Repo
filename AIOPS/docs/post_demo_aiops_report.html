MedicalAgent and Prometheus Linkage

MedicalAgent does not push logs to Prometheus. It exposes a Prometheus-compatible metrics endpoint, and Prometheus scrapes it. 

MedicalAgent
  exposes: http://10.169.91.16:8002/metrics
  job label in Prometheus: medical-rag

Prometheus
  endpoint: http://10.169.82.55:9090
  target: http://10.169.91.16:8002/metrics
  state: UP

RCA / Investigation Flow
  reads Prometheus API: http://10.169.82.55:9090/api/v1/query_range
  converts metric samples into evidence rows displayed as source=prometheus

The displayed rows such as medical_rag_llm_current_window_hits=12 are not raw Prometheus logs. They are RCA evidence entries generated from PromQL results.

Important Metrics

Metric Meaning Source of Value
medical_rag_query_duration_seconds Histogram of MedicalAgent query latency Measured around real /api/query requests
medical_rag_query_requests_total Successful and failed query count Incremented by MedicalAgent per request outcome
medical_rag_query_in_flight Current concurrent MedicalAgent query requests Runtime gauge maintained by MedicalAgent
medical_rag_query_max_concurrency Maximum observed concurrency since process start Runtime gauge maintained by MedicalAgent
medical_rag_llm_requests_total LLM request outcomes by status, deployment, model, and scenario Incremented after Azure OpenAI call success, provider rate limit, or provider error
medical_rag_llm_rate_limit_current_window_hits Number of LLM scenario calls seen in the active rolling 60-second window Calculated in MedicalAgent process memory from actual call timestamps
medical_rag_llm_rate_limit_per_minute Configured request-per-minute limit for the low-rate deployment scenario Read from MedicalAgent config, exported as a gauge for RCA context
medical_rag_llm_rate_limit_remaining Remaining requests in the local observed rolling window Calculated as configured limit minus current observed window hits

How RCA Receives Prometheus Data

  1. The dashboard issue references an application, trace, rule, and incident timestamp.
  2. The RCA investigation service calls Prometheus at http://10.169.82.55:9090.
  3. It runs PromQL queries for latency, concurrency, query errors, LLM request totals, and LLM rate-limit metrics.
  4. The Prometheus response is transformed into readable evidence rows with source=prometheus.
  5. The RCA agent combines those rows with trace/span evidence, including the failed openai_generation span for provider errors.
Example PromQL used by RCA:

max(medical_rag_llm_rate_limit_current_window_hits{app="medical-rag"}) by (scenario, deployment)
max(medical_rag_llm_rate_limit_per_minute{app="medical-rag"}) by (scenario, deployment)
sum(increase(medical_rag_llm_requests_total{app="medical-rag",status="rate_limited"}[5m])) by (scenario, deployment, model)
histogram_quantile(0.95, sum(rate(medical_rag_query_duration_seconds_bucket{app="medical-rag"}[5m])) by (le))

Current Limitations

  • Prometheus captures numeric metrics, not application log lines. Detailed error text still comes from traces/spans.
  • Runtime gauges such as current rolling-window hits reset when the MedicalAgent process restarts. Prometheus retains scraped historical samples, but the live application memory starts fresh.
  • Prometheus increase() may return fractional values because it extrapolates between scrape intervals. RCA display should round human-facing count evidence where appropriate.
  • The Azure OpenAI rate-limit scenario depends on the provider actually returning a throttle response. If Azure allows burst traffic, the incident may not occur exactly on request 11.
  • The MedicalAgent metrics exporter is dependency-light and process-local. In production, use a standard metrics library or OpenTelemetry collector pattern if multi-process workers are used.
  • For multiple replicas, per-pod in-memory counters must be aggregated in Prometheus. A single pod's current-window gauge should not be treated as global cluster truth unless the query sums or maxes across pods intentionally.
  • Remediation can plan infrastructure changes, but actual cluster changes require Kubernetes/cloud permissions.

Can MedicalAgent Be Hosted as a Pod?

Yes. The MedicalAgent can be hosted as a Kubernetes pod and tested under real concurrent traffic. This is the preferred production-like validation for the load scenario.

Recommended Kubernetes Setup

  • Create a Deployment for MedicalAgent with CPU and memory requests/limits.
  • Expose it with a Service on port 8002.
  • Add Prometheus scrape annotations or a ServiceMonitor for /metrics.
  • Store API keys in Kubernetes Secret, not in container images.
  • Use readiness and liveness probes on /api/health.

Concurrency Test Options

  • Run k6, Locust, or hey against the service endpoint.
  • Scale replicas from 1 to N and compare p95 latency, error rate, and max concurrency.
  • Set low CPU/memory limits intentionally in a test namespace to demonstrate saturation.
  • Let Prometheus capture pod CPU, memory, request latency, HTTP errors, and MedicalAgent custom metrics.
Example validation target:

http://medical-rag.<namespace>.svc.cluster.local:8002/api/query

Key Prometheus signals:
up{job="medical-rag"}
medical_rag_query_in_flight{job="medical-rag"}
medical_rag_query_max_concurrency{job="medical-rag"}
medical_rag_query_requests_total{job="medical-rag",status="error"}
histogram_quantile(0.95, sum(rate(medical_rag_query_duration_seconds_bucket{job="medical-rag"}[5m])) by (le))

This pod-based test can produce a realistic incident where user traffic overwhelms the service, Prometheus captures the degraded metrics, AIOps raises an issue, RCA identifies resource or concurrency pressure, and remediation recommends scaling, resource tuning, or code-level backpressure.
